## 提示词工程

### 结构化提示词

使用清晰的结构描述任务：

```markdown
## 任务
重构用户认证模块

## 背景
- 当前使用 session 认证
- 需要迁移到 JWT
- 保持向后兼容

## 要求
1. 使用 RS256 算法
2. Token 有效期 24 小时
3. 支持刷新 Token
4. 添加速率限制

## 约束
- 不要破坏现有 API
- 保持测试覆盖率 > 80%
- 符合项目代码规范

## 输出
1. 架构设计文档
2. 核心代码实现
3. 迁移指南
4. 测试用例
```

### 上下文提供

提供足够的上下文信息：

```
项目技术栈：
- 前端：React 18 + TypeScript
- 后端：Node.js + Express
- 数据库：PostgreSQL
- 认证：当前使用 express-session

相关文件：
- src/auth/login.ts
- src/auth/middleware.ts
- tests/auth/login.test.ts
```

### 迭代式开发

将大任务分解为多轮对话：

```
第一轮：分析需求
> 分析现有的用户认证流程，列出需要修改的文件

第二轮：设计方案
> 基于分析结果，设计 JWT 迁移方案

第三轮：实现代码
> 实现核心的 JWT 认证中间件

第四轮：测试验证
> 添加单元测试，覆盖边界情况
```

### 明确约束条件

清楚说明限制和边界：

```
约束条件：
- 不要修改数据库 schema
- 保持 API 响应格式不变
- 函数体不超过 50 行
- 不使用第三方库，使用内置模块
```

## 代码审查

### 使用 code-review 插件

```bash
# 标准审查
/code-review

# 添加评论到 PR
/code-review --comment

# 调整置信度阈值
/code-review --confidence 90
```

### 审查关注点

| 方面 | 检查项 |
|------|--------|
| **规范合规** | CLAUDE.md、命名规范、格式化 |
| **代码质量** | 复杂度、重复代码、魔法数字 |
| **Bug 检测** | 空指针、资源泄漏、边界条件 |
| **测试覆盖** | 单元测试、边界测试、异常测试 |
| **安全性** | 注入漏洞、敏感数据、权限检查 |

### 审查反馈处理

1. **优先处理 Critical 问题**
   - 安全漏洞
   - 功能 Bug
   - 资源泄漏

2. **评估 Major 问题**
   - 代码复杂度
   - 可维护性问题
   - 性能瓶颈

3. **选择性处理 Minor 问题**
   - 格式化
   - 注释风格
   - 命名建议

### 避免误报

- 置信度 < 80% 的反馈谨慎对待
- 结合项目实际情况判断
- 必要时人工复核

## Git 工作流

### 分支管理

```bash
# 功能开发
git checkout -b feature/<功能简述>-<日期>

# Bug 修复
git checkout -b fix/<问题简述>

# 紧急修复
git checkout -b hotfix/<问题简述>
```

### 提交规范

遵循 Conventional Commits：

```
<type>(<scope>): <subject>

<body>

<footer>
```

**Type 列表**:
| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 文档更新 |
| `refactor` | 重构 |
| `test` | 测试 |
| `chore` | 构建/工具 |

**示例**:
```
feat(auth): 新增 JWT 认证支持

- 实现 JWT 生成和验证
- 添加刷新 Token 机制
- 更新认证中间件

Closes #42
```

### 使用自动化命令

```bash
# 一键完成提交、推送、创建 PR
/commit-push-pr

# Issue 自动分类
/triage-issue

# Issue 去重
/dedupe
```

### PR 最佳实践

1. **PR 大小**: 控制在 400 行以内
2. **描述清晰**: 说明改动原因和影响
3. **关联 Issue**: 使用 `Closes #编号` 自动关闭
4. **审查请求**: 至少请求 1 人审查

## 插件使用

### 必装插件

#### feature-dev - 功能开发工作流

```bash
/feature-dev <功能描述>
```

**7 阶段流程**:
1. 发现 - 澄清需求
2. 探索 - 分析代码库
3. 提问 - 解决模糊点
4. 设计 - 提供架构方案
5. 实现 - 编写代码
6. 审查 - 质量检查
7. 总结 - 记录完成内容

#### code-review - 代码审查

```bash
/code-review
```

**5 个并行 Agent**:
- CLAUDE.md 合规检查
- Bug 扫描
- 历史上下文分析
- PR 历史分析
- 代码注释检查

#### hookify - 自定义 Hooks

```bash
# 创建规则
/hookify 警告我使用 console.log

# 列出规则
/hookify:list

# 配置规则
/hookify:configure
```

### 插件组合使用

```bash
# 功能开发完整流程
/feature-dev 添加用户注销功能
# → 实现完成后
/code-review
# → 审查通过后
/commit-push-pr
```

## 安全配置

### 权限控制

#### 个人开发（宽松）

```json
{
  "permissions": {
    "disableBypassPermissionsMode": "disable"
  }
}
```

#### 团队项目（标准）

```json
{
  "permissions": {
    "ask": ["Bash"],
    "deny": ["WebSearch"]
  }
}
```

#### 企业环境（严格）

```json
{
  "permissions": {
    "ask": ["Bash"],
    "deny": ["WebSearch", "WebFetch"]
  },
  "allowManagedPermissionRulesOnly": true,
  "allowManagedHooksOnly": true
}
```

### 沙箱配置

```json
{
  "sandbox": {
    "enabled": true,
    "autoAllowBashIfSandboxed": false,
    "network": {
      "allowLocalBinding": false,
      "allowedDomains": ["api.example.com"]
    }
  }
}
```

### 危险命令拦截

使用 hookify 创建规则：

```bash
/hookify 阻止我执行 rm -rf /
/hookify 警告我使用 eval()
/hookify 阻止我提交 .env 文件
```

## 性能优化

### Token 使用优化

#### 减少上下文

```bash
# 压缩会话历史
/compact

# 清除无关上下文
/clear
```

#### 文件加载策略

| 场景 | 策略 |
|------|------|
| 小项目 | 全量加载 |
| 大项目 | 按需加载关键文件 |
| 单文件分析 | 只加载目标文件 |

### 响应速度优化

#### 模型选择

| 任务 | 模型 | 响应时间 |
|------|------|----------|
| 简单问答 | Haiku | ~2 秒 |
| 代码生成 | Sonnet | ~10 秒 |
| 架构设计 | Opus | ~30 秒 |

#### 任务分解

将大任务拆分为小步骤：

```
❌ 重构整个认证模块
✅ 第一步：分析现有代码
✅ 第二步：设计新架构
✅ 第三步：实现核心函数
✅ 第四步：添加测试
```

### 缓存利用

- 重复问题使用相同会话
- 避免频繁切换项目
- 使用 `/resume` 恢复会话
