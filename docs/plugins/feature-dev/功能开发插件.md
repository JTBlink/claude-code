# feature-dev 插件

> 7 阶段结构化功能开发工作流

## 功能说明

`feature-dev` 插件提供了一个全面的、结构化的功能开发工作流，通过 7 个明确的阶段引导 Claude Code 完成从需求理解到代码实现的全过程。

## 安装方式

```bash
# 插件已包含在项目中
# 直接在会话中使用 /feature-dev 命令
```

## 命令参考

### `/feature-dev <功能描述>`

启动 7 阶段功能开发工作流。

**使用方式**:
```bash
/feature-dev 添加 OAuth 用户认证功能
/feature-dev 实现用户注销功能
/feature-dev 添加密码重置功能
```

**功能描述建议**:
- 清晰描述要实现的功能
- 包含关键需求
- 指定技术栈（如适用）

## 7 阶段流程

### 阶段 1: 发现 (Discovery)

**目标**: 澄清需求，理解要构建什么

**活动**:
- 确认功能目标
- 识别利益相关者
- 明确成功标准
- 确定优先级

**输出**:
```markdown
## 功能需求文档

### 目标
<功能要解决什么问题>

### 用户故事
- 作为 <角色>，我希望 <功能>，以便 <价值>

### 成功标准
- [ ] 标准 1
- [ ] 标准 2

### 优先级
P0: 必须实现
P1: 应该实现
P2: 可以实现
```

**示例对话**:
```
用户：/feature-dev 添加用户登录功能

Claude: 让我先了解一些细节：

1. 支持哪些登录方式？
   - 邮箱密码
   - 第三方 OAuth (GitHub, Google)?
   - 手机短信？

2. 需要什么安全特性？
   - 双因素认证
   - 密码强度检查
   - 登录尝试限制？

3. 有设计稿吗？
```

### 阶段 4: 设计 (Design)

**目标**: 提供多个架构方案

**活动**:
- 设计 API 接口
- 设计数据模型
- 设计流程
- 评估方案

**输出**:
```markdown

### 方案 A: JWT 认证

**优点**:
- 无状态，易于扩展

- 移动端友好

**缺点**:
```

### 阶段 5: 实现 (Implementation)

**目标**: 编写代码

**活动**:
- 创建文件结构

**输出**:
```markdown

### 已完成
- [x] 创建 User 模型

- [x] 编写单元测试

### 待完成
- [ ] 集成测试
- [ ] API 文档
```

**代码示例**:
```python
# src/auth/login.py
from flask import Blueprint, request, jsonify
from flask_jwt_extended import create_access_token, jwt_required
from werkzeug.security import check_password_hash
from models.user import User
from extensions import db

auth_bp = Blueprint('auth', __name__)

@auth_bp.route('/login', methods=['POST'])
def login():
    data = request.get_json()
    email = data.get('email')
    password = data.get('password')
    
    user = User.query.filter_by(email=email).first()
    if not user or not check_password_hash(user.password, password):
        return jsonify({'error': 'Invalid credentials'}), 401
    
    access_token = create_access_token(identity=user.id)
    return jsonify({
        'access_token': access_token,
        'expires_in': 3600
    })
```

## 使用示例

### 示例 1: 添加用户认证

```bash
/feature-dev 添加 GitHub OAuth 登录功能
```

**流程**:
1. **发现**: 确认支持 GitHub OAuth，需要获取用户邮箱
2. **探索**: 分析现有 auth 模块，使用 flask-dance
3. **提问**: 确认 Callback URL、权限范围
4. **设计**: 设计 OAuth 流程和用户绑定逻辑

## 最佳实践

### 阶段切换

- **完成标准**: 每个阶段有明确的输出和完成标准
- **用户确认**: 关键阶段需要用户确认后再继续
- **灵活调整**: 根据项目复杂度调整阶段深度

### 沟通技巧

- **清晰表达**: 明确当前所处阶段和目标
- **主动提问**: 发现模糊点及时确认
- **文档记录**: 每个阶段输出文档便于追溯

### 质量保证

- **测试先行**: 实现前编写测试用例
- **持续审查**: 每个阶段结束后审查
- **迭代改进**: 根据反馈调整后续阶段

## 故障排查

### 问题：阶段卡住

**原因**: 信息不足或需求模糊

**解决方案**:
```bash
# 回到提问阶段
让我们回到阶段 3，我需要确认一些细节...

# 或请求用户提供更多信息
请提供更多关于 XX 的信息...
```

### 问题：阶段跳跃

**原因**: 用户急于看到代码

**解决方案**:
```markdown
我理解您想尽快看到代码，但先完成前面阶段可以：
1. 避免返工
2. 确保实现正确
3. 提高代码质量

让我们先完成阶段 2 的探索...
```

### 问题：范围蔓延

**原因**: 需求不断扩展

**解决方案**:
```markdown
我注意到需求有所扩展。建议：
1. 当前功能按原计划完成
2. 新增需求记录为后续迭代
3. 优先级排序后实现
```
